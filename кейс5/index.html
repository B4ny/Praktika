Эпизод III
Анализ проекта Django

Кейс №3 представляет собой веб-приложение на Django для приветствия пользователя по имени. Проект состоит из Django-проекта «кейс3_new», приложения «greetings» с моделью Name, формой NameForm, представлением index, HTML-шаблона с CSS-стилизацией и базы данных SQLite.

Настало время провести детальный анализ проделанной работы по девяти ключевым критериям качества программного обеспечения и предложить пути улучшения...

1. Функциональность

Что реализовано:

  • Форма для ввода имени пользователя
  • Сохранение введённых имён в базу данных
  • Отображение персонализированного приветствия
  • Валидация на пустое поле и CSRF-защита
  • Админ-панель Django для управления записями

Проблемы:

  • Отсутствие уникальности имён — дубли в БД
  • Нет отображения истории введённых имён
  • Ограниченная функциональность — только одна операция
# models.py — добавить уникальность class Name(models.Model): name = models.CharField(max_length=100, unique=True) created_at = models.DateTimeField(auto_now_add=True)

2. Производительность

Что хорошо:

  • Использование индексов SQLite
  • Минимальное количество запросов к БД
  • Лёгкий CSS без тяжёлых изображений

Проблемы:

  • Рост таблицы без ограничений — замедление запросов
  • Отсутствие пагинации
  • DEBUG=True в production сильно замедляет работу
from django.core.paginator import Paginator paginator = Paginator(Name.objects.all(), 20) page = request.GET.get('page') names = paginator.get_page(page)

3. Удобство использования

Что хорошо:

  • Адаптивный дизайн с CSS Flexbox
  • Приятная цветовая гамма и градиентный фон
  • Hover-эффекты и фокус-стили

Проблемы:

  • Кнопка «Submit» на английском — несоответствие интерфейсу
  • Нет подтверждения успешного сохранения
  • Отсутствие счётчика посещений/имён

4. Безопасность

Что хорошо:

  • CSRF-защита включена
  • Django-шаблонизатор экранирует вывод (XSS-защита)
  • XFrameOptionsMiddleware и SecurityMiddleware подключены

Критические проблемы:

  • SECRET_KEY захардкожен — серьёзная уязвимость
  • DEBUG=True — раскрывает детали ошибок
  • ALLOWED_HOSTS пустой — уязвимость для Host header attacks
  • Нет rate limiting — возможен DoS
  • Пароль админа admin123 — легко подбирается
SECRET_KEY = os.environ.get('DJANGO_SECRET_KEY') DEBUG = os.environ.get('DJANGO_DEBUG', 'False') == 'True' ALLOWED_HOSTS = ['yourdomain.com', 'localhost']

5. Масштабируемость

Что хорошо:

  • Стандартная архитектура Django (MTV)
  • Разделение на приложения
  • Модельная структура БД легко расширяется

Проблемы:

  • SQLite — не подходит для высоких нагрузок
  • Отсутствие кэширования
  • Нет CDN для статических файлов
  • Монолитная архитектура
# Переход на PostgreSQL DATABASES = { 'default': { 'ENGINE': 'django.db.backends.postgresql', 'NAME': os.environ.get('DB_NAME'), # ... } }

6. Сопровождаемость

Что хорошо:

  • Чистая структура Django-проекта
  • Использование стандартных паттернов
  • CSS вынесен в отдельный файл

Проблемы:

  • Нет requirements.txt — неизвестны версии библиотек
  • Нет docstrings и комментариев
  • Нет .gitignore
  • Нет CI/CD конфигурации

7. Переносимость

Что хорошо:

  • Стандартный Django — работает на любой ОС
  • Использование pathlib.Path

Проблемы:

  • Хардкод путей — нет env-конфигурации
  • Нет Docker-контейнеризации
  • Нет CI/CD для тестирования на разных платформах

8. Качество кода

Что хорошо:

  • Соблюдение PEP 8
  • Читаемые имена переменных
  • Корректное разделение ответственности

Проблемы:

  • Пустой tests.py — нет тестов вообще
  • Нет type hints
  • Валидация в представлении вместо формы
  • Нет обработки исключений

9. Тестирование

Критические проблемы:

  • Полное отсутствие тестов — критическая проблема
  • Нет unit-тестов, интеграционных тестов
  • Нет тестов на валидацию и безопасность
  • Нет нагрузочного тестирования
  • Нет coverage-анализа
class IndexViewTest(TestCase): def test_post_valid_name(self): response = self.client.post( reverse('index'), {'name': 'Иван'} ) self.assertEqual(response.status_code, 200) self.assertContains(response, 'Привет, Иван!')

Приоритетные рекомендации

🔴 Критический приоритет

  • Вынести SECRET_KEY в переменные окружения
  • Установить DEBUG=False для production
  • Заполнить ALLOWED_HOSTS
  • Добавить базовые unit-тесты

🟡 Высокий приоритет

  • Добавить валидацию в формы
  • Сделать имена уникальными
  • Добавить requirements.txt и .gitignore

🟢 Средний приоритет

  • Создать Dockerfile
  • Перейти на PostgreSQL
  • Настроить Redis для кэширования

Заключение

Проект Кейса №3 демонстрирует базовые навыки работы с Django. Приложение выполняет свою основную функцию — приветствие пользователя по имени. Однако проект находится на начальном уровне развития и требует значительной доработки по всем критериям, особенно в области безопасности, тестирования и подготовки к production-развёртыванию.

Реализация предложенных улучшений превратит простое учебное приложение в production-ready проект, соответствующий современным стандартам веб-разработки.
