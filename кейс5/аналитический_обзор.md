# Аналитический обзор проделанной работы в Кейсе №3

## Общая информация о проекте

Кейс №3 представляет собой веб-приложение на Django для приветствия пользователя по имени. Проект состоит из:
- Django-проекта `кейс3_new`
- Приложения `greetings` с моделью `Name`, формой `NameForm`, представлением `index`
- HTML-шаблона с CSS-стилизацией
- Базы данных SQLite

---

## 1. Функциональность

### Что реализовано:
- Форма для ввода имени пользователя
- Сохранение введённых имён в базу данных
- Отображение персонализированного приветствия
- Валидация на пустое поле
- CSRF-защита формы
- Админ-панель Django для управления записями

### Выявленные проблемы:
- **Отсутствие уникальности имён**: одинаковые имена сохраняются многократно, создавая дубли в БД
- **Отсутствие отображения истории**: пользователь не видит ранее введённые имена
- **Нет обработки специальных символов**: возможен ввод HTML/JS-кода (XSS-уязвимость через отображение в шаблоне, хотя Django-шаблонизатор экранирует вывод по умолчанию)
- **Ограниченная функциональность**: приложение выполняет только одну простую операцию

### Решение:

## 2. Производительность

### Что хорошо:
- Использование индексов SQLite (id создаётся автоматически)
- Минимальное количество запросов к БД
- Лёгкий CSS без тяжёлых изображений
- Отсутствие сложных вычислений

### Выявленные проблемы:
- **Рост таблицы без ограничений**: при большом количестве записей запросы будут замедляться
- **Отсутствие пагинации**: при выводе списка всех записей страница будет загружаться долго
- **N+1 проблема потенциальна**: если добавить связанные модели без select_related/prefetch_related
- **DEBUG=True в production**: сильно замедляет работу

### Решение:
```python
# Добавить пагинацию
from django.core.paginator import Paginator

def index(request):
    # ...
    recent_names = Name.objects.all()[:10]  # Ограничить выборку
    # Или с пагинацией:
    paginator = Paginator(Name.objects.all(), 20)
    page = request.GET.get('page')
    names = paginator.get_page(page)
```

## 4. Безопасность

### Что хорошо:
- CSRF-защита включена (`{% csrf_token %}`)
- Django-шаблонизатор автоматически экранирует вывод (защита от XSS)
- XFrameOptionsMiddleware включён (защита от clickjacking)
- SecurityMiddleware подключён
- Парольная валидация настроена

### Выявленные проблемы:
- **SECRET_KEY захардкожен**: `'django-insecure-(qnh##hu2ud$7p-03nrr_qzzna9ipc=jn#fs&$(!%p+x0dxw0s'` — серьёзная уязвимость для production
- **DEBUG=True**: в production показывает детали ошибок и структуру проекта
- **ALLOWED_HOSTS пустой**: уязвимость для HTTP Host header attacks
- **Отсутствие валидации ввода**: возможен ввод очень длинных строк, спецсимволов, SQL-инъекций (хотя ORM защищает)
- **Нет rate limiting**: возможен спам запросами (DoS)
- **Пароль админа простой**: `admin123` — легко подбирается
- **Нет HTTPS-настроек**: отсутствуют SECURE_SSL_REDIRECT, SESSION_COOKIE_SECURE
- **Отсутствие логирования**: невозможно отследить подозрительную активность

### Решение:
```python
# settings.py
import os
from pathlib import Path

SECRET_KEY = os.environ.get('DJANGO_SECRET_KEY')
DEBUG = os.environ.get('DJANGO_DEBUG', 'False') == 'True'

# Безопасность
SECURE_SSL_REDIRECT = True

SECURE_BROWSER_XSS_FILTER = True
SECURE_CONTENT_TYPE_NOSNIFF = True
X_FRAME_OPTIONS = 'DENY'

# Rate limiting

@ratelimit(key='ip', rate='10/m')

    # ...
```

## 8. Качество кода

### Что хорошо:
- Соблюдение PEP 8 (в основном)
- Читаемые имена переменных (`greeting_name`, `error_message`, `entered_name`)
- Использование Django-форм вместо ручной обработки
- Корректное разделение ответственности (models, forms, views)
- Правильное использование Django ORM

### Выявленные проблемы:
- **Пустой tests.py**: совершенно отсутствуют тесты
- **Нет type hints**: сложнее понять типы данных
- **Валидация в представлении, а не в форме**: логика проверки размазана
- **Магические строки**: `'Имя не может быть пустым.'` захардкожены
- **Нет обработки исключений**: если БД недоступна — приложение упадёт
- **Вьюха слишком толстая**: делает слишком много (валидация, сохранение, рендеринг)
- **Нет использования Class-Based Views**: при росте функциональности функциональные вьюхи разрастутся
- **Отсутствие менеджеров моделей**: логика запросов к БД размазана по вьюхам

### Решение:
```python
# forms.py - перенести валидацию
from django import forms
from django.utils.translation import gettext_lazy as _
from .models import Name

class NameForm(forms.Form):
    name = forms.CharField(
        max_length=100,
        label=_('Ваше имя'),
        widget=forms.TextInput(attrs={
            'placeholder': _('Введите ваше имя'),
            'class': 'form-input',
        }),
    )
    
    def clean_name(self) -> str:
        name = self.cleaned_data['name'].strip()
        if not name:
            raise forms.ValidationError(_('Имя не может быть пустым.'))
        if Name.objects.filter(name__iexact=name).exists():
            raise forms.ValidationError(_('Это имя уже существует.'))
        return name

# models.py - добавить менеджер
class NameManager(models.Manager):
    def get_recent(self, limit: int = 10):
        return self.order_by('-id')[:limit]
    
    def get_unique_count(self) -> int:
        return self.values('name').distinct().count()

class Name(models.Model):
    name = models.CharField(max_length=100, unique=True)
    created_at = models.DateTimeField(auto_now_add=True)
    
    objects = NameManager()
```

## 9. Тестирование

### Что хорошо:
- Django TestCase импортирован (хотя и не используется)
- Структура проекта позволяет легко добавить тесты

### Выявленные проблемы:
- **Полное отсутствие тестов**: tests.py пустой
- **Нет unit-тестов**: не проверяется логика форм, моделей, вьюх
- **Нет интеграционных тестов**: не проверяется полный цикл запроса
- **Нет тестов на валидацию**: не проверяется обработка пустых строк, спецсимволов
- **Нет нагрузочного тестирования**: неизвестно, как поведёт себя приложение под нагрузкой
- **Нет тестов безопасности**: CSRF, XSS, SQL-инъекции не проверяются
- **Нет coverage-анализа**: неизвестно, какой процент кода покрыт тестами

### Решение:
```python
# tests.py
from django.test import TestCase, Client
from django.urls import reverse
from .models import Name
from .forms import NameForm


class NameModelTest(TestCase):
    def test_name_creation(self):
        name = Name.objects.create(name='Иван')
        self.assertEqual(str(name), 'Иван')
        self.assertEqual(Name.objects.count(), 1)
    
    def test_name_unique_constraint(self):
        Name.objects.create(name='Иван')
        with self.assertRaises(Exception):
            Name.objects.create(name='Иван')


class NameFormTest(TestCase):
    def test_valid_form(self):
        form = NameForm(data={'name': 'Иван'})
        self.assertTrue(form.is_valid())
    
    def test_empty_name(self):
        form = NameForm(data={'name': '   '})
        self.assertFalse(form.is_valid())
    
    def test_name_too_long(self):
        form = NameForm(data={'name': 'а' * 101})
        self.assertFalse(form.is_valid())
    
    def test_html_injection_attempt(self):
        form = NameForm(data={'name': '<script>alert("XSS")</script>'})
        # Django автоматически экранирует, но форма может быть валидной
        self.assertTrue(form.is_valid())


class IndexViewTest(TestCase):
    def setUp(self):
        self.client = Client()
    
    def test_get_request(self):
        response = self.client.get(reverse('index'))
        self.assertEqual(response.status_code, 200)
        self.assertTemplateUsed(response, 'greetings/index.html')
        self.assertContains(response, 'Добро пожаловать!')
    
    def test_post_valid_name(self):

        self.assertEqual(response.status_code, 200)
        self.assertContains(response, 'Привет, Иван!')
        self.assertEqual(Name.objects.count(), 1)
    
    def test_post_empty_name(self):
        response = self.client.post(reverse('index'), {'name': ''})
        self.assertEqual(response.status_code, 200)
        self.assertContains(response, 'не может быть пустым')
    
    def test_post_duplicate_name(self):
        Name.objects.create(name='Иван')
        response = self.client.post(reverse('index'), {'name': 'Иван'})
        self.assertEqual(response.status_code, 200)
        self.assertEqual(Name.objects.count(), 1)
    
    def test_csrf_protection(self):
        response = self.client.post(reverse('index'), {'name': 'Иван'})
        # Без CSRF-токена запрос должен быть отклонён
        # Но Client автоматически обрабатывает CSRF
        pass


class SecurityTest(TestCase):
    def test_xss_protection(self):
        malicious_input = '<script>alert("XSS")</script>'
        response = self.client.post(reverse('index'), {'name': malicious_input})
        # Проверяем, что скрипт не выполняется
```

## Заключение

Проект Кейса №3 демонстрирует базовые навыки работы с Django: создание моделей, форм, представлений и шаблонов. Приложение выполняет свою основную функцию — приветствие пользователя по имени. Однако проект находится на начальном уровне развития и требует значительной доработки по всем критериям, особенно в области безопасности, тестирования и подготовки к production-развёртыванию.

Реализация предложенных улучшений превратит простое учебное приложение в production-ready проект, соответствующий современным стандартам веб-разработки.
